# Project Map --- Programmatic Content Site

This document explains the directory architecture and responsibilities
of the repository.

The architecture intentionally separates content generation, site
planning, and frontend rendering.

------------------------------------------------------------------------

# Architecture Overview

```
content/data/  →  scripts/compute-pages.ts  →  content/index/page-definitions.json
                                                         ↓
                                             scripts/generate-pages.ts
                                                         ↓
                                                  content/pages/*.md
                                                         ↓
                                                     apps/web/
```

The key rule: the frontend renders content but does not generate it.

------------------------------------------------------------------------

# Top-Level Directory Map

```
content/    structured datasets + generated artifacts
scripts/    pipeline scripts
src/        shared libraries (lib/ + types/)
apps/       frontend application
docs/       architecture documentation
```

------------------------------------------------------------------------

# content/

## content/data/

Structured datasets. The source of truth for all entities.

```
content/data/
  tools/
    tools.json           # Tool definitions (id, slug, name, categories, audiences, etc.)
  taxonomy/
    categories.json      # Content categories (writing, video, productivity, transcription)
    audiences.json       # Target audiences (students, marketers, creators, startups)
    use-cases.json       # Use cases (blog-writing, note-taking, video-ads, meeting-notes)
    features.json        # AI tool features (ai-suggestions, templates, collaboration, etc.)
    price-tiers.json     # Pricing tiers (free, freemium, premium)
    integrations.json    # Integration targets (slack, zapier, google-drive, zoom)
```

Rules:
- must remain structured JSON — no prose paragraphs
- is the source of truth for all entity data
- tool fields: id, name, slug, description, tagline, website, categories[], audiences[],
  useCases[], features[], priceTiers[], integrations[], alternatives[]

## content/index/

Artifacts produced by `compute-pages.ts`. Represents the canonical site plan.
These files are gitignored (generated artifacts).

```
content/index/
  page-definitions.json          # Full page index: all candidates (valid + rejected)
  page-definition-report.json    # Summary report: counts, rejection reasons, warnings
```

## content/pages/

Generated markdown files, one per valid page definition.
These files are gitignored (safe to regenerate).

Examples: `best-writing-tools.md`, `jasper-alternatives.md`, `copy-ai-vs-notion.md`

Rules:
- generated by `generate-pages.ts` — do not manually edit
- each file contains YAML frontmatter + deterministic sections + LLM placeholder comments
- safe to delete and regenerate via `pnpm generate-pages`

------------------------------------------------------------------------

# scripts/

Pipeline scripts. Each runs a single stage of the content engine.

```
scripts/
  compute-pages.ts    # Stage 1: compute page definitions + write index
  generate-pages.ts   # Stage 2: render valid definitions to markdown
  qa-check.ts         # Stage 3: validate definitions + generated files
```

Pipeline: `compute-pages` → `generate-pages` → `qa-check`

Scripts must not contain frontend logic. They depend on `src/lib/` and `src/types/`.

------------------------------------------------------------------------

# src/

Shared TypeScript libraries used by the pipeline scripts.

## src/lib/

```
src/lib/
  loadData.ts       # Reads all JSON datasets from content/data/ → Dataset
  slugify.ts        # Converts strings to URL-safe slugs
  pageRules.ts      # 9 rule functions that emit PageDefinition candidates
  pageBuilders.ts   # Constructs PageDefinition objects from rule parameters
  pageIndex.ts      # Validate, deduplicate, build, persist, and load the page index
  markdown.ts       # Renders PageDefinition → markdown string (frontmatter + sections)
  llm.ts            # generateNarrative() — currently returns LLM_PLACEHOLDER comments
  validate.ts       # QA checks: duplicates, tool counts, frontmatter, file presence
```

## src/types/

```
src/types/
  entities.ts    # Tool, Category, Audience, UseCase, Feature, PriceTier, Integration, Dataset
  pages.ts       # PageType, SectionName, PageDefinition, PageIndex, PageDefinitionReport,
                 # ValidationResult, ValidationReport
```

Key types:
- `PageType` — 9 variants: category, audience-category, use-case, audience-use-case,
  feature, pricing, alternatives, comparison, tool-detail
- `SectionName` — 6 variants: introduction, toolList, comparisonTable, bestFor,
  prosAndCons, faq
- `PageDefinition` — slug, pageType, title, description, canonicalKey, sourceRule,
  entities, matchedToolIds, supportCount, sections, isValid, warnings, rejectionReason,
  overlapScore

# apps/

Frontend application(s). Currently not yet built (Phase 3 in build-sequence.md).

```
apps/

```

When built, responsibilities:
- load markdown pages from `content/pages/`
- map slug → route
- render markdown content
- return 404 for unknown slugs
- expose page metadata

The frontend must never compute page definitions.

# Data Flow Summary

```
content/data/*.json
        ↓  loadData()
   Dataset (tools, categories, audiences, ...)
        ↓  applyAllRules()
   PageDefinition[] candidates (all, including invalid)
        ↓  validateCandidates() + deduplicateCandidates()
   PageDefinition[] (isValid=true/false set)
        ↓  buildPageIndex() + writePageIndex()
   content/index/page-definitions.json
        ↓  loadPageIndex() + renderMarkdown()
   content/pages/{slug}.md (one per valid page)
        ↓  apps/web/ (Phase 3+)
   rendered website
```

------------------------------------------------------------------------

# Key Principle

The content engine defines what pages exist. The frontend defines how
pages are displayed.
