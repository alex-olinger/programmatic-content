# Project Map --- Programmatic Content Site

This document explains the directory architecture and responsibilities
of the repository.

The architecture intentionally separates content generation, site
planning, and frontend rendering.

------------------------------------------------------------------------

# Architecture Overview

datasets → page-definition computation → site plan artifact → content
generation → frontend rendering

The key rule: The frontend renders content but does not generate it.

------------------------------------------------------------------------

# Top-Level Directory Map

content/ scripts/ src/ apps/ docs/

------------------------------------------------------------------------

# content/

Stores generated artifacts and structured datasets.

content/ data/ templates/ pages/ index/

## content/data/

Structured datasets used by the generator.

Examples: tools/ categories/ audiences/ use_cases/ features/ pricing/

Rules: - must remain structured data - should contain no prose
paragraphs - is the source of truth for entities

## content/templates/

Templates used by the content generator.

Typical sections: intro tool shortlist comparison table best-for section
faq

## content/pages/

Generated markdown pages.

Examples: best-ai-writing-tools-for-students.md jasper-alternatives.md
notion-vs-obsidian.md

Rules: - generated by scripts - should not be manually edited - safe to
regenerate

## content/index/

Artifacts produced during page-definition computation.

Examples: page-definitions.json site-plan-summary.json

These represent the canonical site plan.

------------------------------------------------------------------------

# scripts/

Scripts responsible for running the content engine pipeline.

Typical scripts: compute-pages.ts generate-pages.ts qa-check.ts

Pipeline flow: compute-pages → generate-pages → qa-check

Scripts must not contain frontend logic.

------------------------------------------------------------------------

# src/

Shared libraries used by the generator.

src/ lib/ types/

## src/lib/

Reusable logic: slug generation page rules dataset utilities validation
helpers

## src/types/

TypeScript types used across the engine.

Examples: Tool Category Audience PageDefinition SitePlan

------------------------------------------------------------------------

# apps/

Frontend applications live here.

apps/ web/

The frontend renders content but does not generate it.

------------------------------------------------------------------------

# apps/web/

The website frontend.

Responsibilities: - load markdown pages - map slug → route - render page
content - provide metadata

The frontend must never compute page definitions.

------------------------------------------------------------------------

# docs/

Architecture documentation and project rules.

Examples: build-sequence.md content-engine-rules.md project-map.md
ai-editing-rules.md

------------------------------------------------------------------------

# Data Flow Summary

datasets ↓ compute page definitions ↓ site plan artifact ↓ generate
markdown pages ↓ frontend renders markdown ↓ users access site

------------------------------------------------------------------------

# Key Principle

The content engine defines what pages exist. The frontend defines how
pages are displayed.
